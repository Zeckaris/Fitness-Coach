# V6.1 — Context & Token Optimization: Findings & Solution Design

## 1. Method

Reviewed 5 Langfuse traces covering a range of interaction complexity:

| Test | Scenario | LLM Calls | Total Tokens |
|---|---|---:|---:|
| 1 | Small talk ("Hello") | 1 | 5,097 |
| 2 | Exercise recommendation (1 tool) | 2 | 11,351 |
| 3 | Injury + plan update (7 tool calls) | 5 | 43,662 |
| 4 | Stateful follow-up (2 short turns) | 1 | 5,333 |
| 6 | Stateful complex request (plan + search) | 4 | 37,308 |

## 2. Findings

### 2.1 There is a large fixed "baseline tax" per LLM call

Test 1 ("Hello", effectively no real content) still costs **5,089 input tokens**. This is almost entirely the static system prompt plus the JSON tool schemas for all 7 bound tools. This cost is paid **in full on every single LLM call**, not just once per session.

### 2.2 Simple multi-turn chat is NOT the problem

Test 4 shows two real conversational turns adding only ~150 tokens over baseline. LangGraph's message history for ordinary chat is cheap and not a priority for optimization right now.

### 2.3 The baseline tax compounds across ReAct tool loops

This is the primary driver of the large totals in Tests 3 and 6. Complex requests trigger mandatory multi-step tool chains (e.g. `get_current_plan → get_past_plans → search_fitness_knowledge_base → search_workout_library → update_three_day_plan`), and **each step is a separate LLM call that re-pays the full ~5,100 token baseline**:

- Test 3: 5 LLM calls × ~5,100 baseline ≈ 25,500 tokens — roughly **58% of the 43,662 total** — before any tool output or real content is counted.
- Test 6: 4 LLM calls, 35,083 input tokens alone, consistent with the same pattern at a larger scale.

## 3. Solution Design

### 3.1 Prompt / context caching

**Problem:** The system prompt + tool schemas are byte-identical on every call within a session, but are billed at full price every time.

**Solution:** Enable prompt caching so the static prefix (system prompt + tool definitions) is cached after the first call in a session and reused at reduced cost on subsequent calls in the same turn/session.

**Expected impact:** Highest ROI of the two fixes. Directly attacks the ~5,100-token baseline that gets re-paid on every LLM call — should cut Test 3/6-style multi-call totals substantially with zero change to agent output quality.

**Next step:** Verify whether `ChatGoogleGenerativeAI` / the Gemini API path currently used in `get_llm()` supports and is actually invoking context caching, and wire it in if not already active.

### 3.2 Shrink the system prompt

**Problem:** `SYSTEM_PROMPT` in `agent/prompts.py` has grown across V4–V6 and now includes some redundancy — notably, tool docstrings restate detail that's already implicit in each tool function's own docstring/schema (which is also sent to the model via tool binding).

**Solution:** Trim redundant/duplicated explanation from the prompt text while preserving the behavior-critical instructions (disruption philosophy, mandatory injury-propagation rule, plan-vs-today boundary, mandatory sequencing for plan updates).

**Expected impact:** Smaller than caching, but reduces the baseline tax itself (not just its repeated cost), so it compounds with caching rather than overlapping.

**Next step:** Line-by-line audit of `SYSTEM_PROMPT` to flag redundant-with-schema content vs. behavior-critical content, then trim only the former.
