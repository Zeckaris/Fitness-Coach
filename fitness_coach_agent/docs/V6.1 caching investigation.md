# V6.1 — Caching Investigation: Findings

## Goal

Determine whether Gemini prompt/context caching could be used to reduce the ~5,100-token static baseline (system prompt + tool schemas) paid on every LLM call, per the solution proposed in the original token audit.

## What We Checked

### 1. Implicit caching (automatic, on by default)

- Gemini's implicit caching is enabled by default for `gemini-3.5-flash` (Gemini 2.5+ family), with a minimum input-token threshold of ~2,048 tokens. The baseline (~5,100 tokens) clears that threshold.
- Ran `tests/check_caching.py` — a diagnostic script that calls the real, tools-bound `get_llm()` with an identical system prompt across multiple consecutive calls and inspects `usage_metadata` for cache hits.
- **Result:** Two consecutive successful calls both returned `cache_read: 0`. No cache hit occurred, despite an identical, static, above-threshold prompt sent seconds apart.

### 2. Explicit caching (manual, via `create_context_cache`)

- Gemini's API rejects any request that combines `cached_content` with `tools`.
- My agent binds 7 tools via `bind_tools()`, which LangChain injects into every generation request — making this combination structurally incompatible, not just difficult to configure.
- Confirmed as a known, still-open issue: `langchain-ai/langchain-google` #1528.

## Corroborating Evidence

The `cache_read: 0` result we observed matches widely reported, independent findings from other developers:

- Google's own developer forum has multiple threads of users reporting implicit caching returning zero cache hits on long, static, repeated prompts.
- Multiple open GitHub issues on `googleapis/python-genai` document inconsistent or entirely absent implicit cache hits under similar conditions, including a reproducible "dead zone" on newer Gemini 3 Flash models.
- Google's own documentation notes implicit caching carries "no cost saving guarantee" — it is opportunistic infrastructure behavior, not a guaranteed mechanism.

## Conclusion

**Caching is not a viable lever for this project right now.** This is a platform-level limitation, not a mistake in my implementation:

- Implicit caching is unreliable in practice and did not hit in my own test, consistent with broad community experience.
- Explicit caching is structurally blocked by the tools-binding incompatibility as long as the agent uses bound tools.

**Decision:** Rule out caching for this pass. Revisit only if Google improves implicit caching reliability, or resolves the explicit-caching/tools incompatibility, in the future.
