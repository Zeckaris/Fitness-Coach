# V6.1 — Validation Results: Prompt & Tool Schema Compaction
 
## 1. Purpose
 
The original V6.1 audit measured the static baseline (system prompt + bound tool schemas)
before and after compaction as an isolated, synthetic measurement, showing a 42.5% reduction
(5,100 → 2,929 tokens).
 
This document validates that reduction against real Langfuse traces by rerunning the original
test scenarios post-compaction. It records the results as observed, including two scenarios
where the rerun did not produce a directly comparable trace to the original baseline.
 
## 2. Method
 
Each of the five original test scenarios (Section reference: V6 token audit) was rerun under
identical or near-identical conditions against the compacted system prompt and tool schemas.
Results are reported per test, with token, tool-call, and latency deltas where the comparison
is valid, and with an explanation where it is not.

## 3. Test 1 — Small Talk
 
**User message:** `Hello`
 
| Metric | Before (V6) | After (V6.1) | Change |
|---|---:|---:|---:|
| Graph Nodes | history_check_node → coach_node | history_check_node → coach_node | — |
| LLM Calls | 1 | 1 | — |
| Tool Calls | 0 | 0 | — |
| Input Tokens | 5,089 | 2,901 | -2,188 |
| Output Tokens | 8 | 17 | +9 |
| **Total Tokens** | **5,097** | **2,918** | **-2,179 (-42.7%)** |
| Latency | 5.51s | 5.40s | -0.11s |
 
**Findings:**
 
- The input token reduction (-2,188) closely matches the static-baseline compaction estimate
  (~2,171 tokens), confirming the estimate holds in a live trace.
- The output token increase (8 → 17) is attributable to `yesterday_context` being populated in
  this trace (a prior check-in existed), which was not the case in the original baseline trace.
  This is `history_check_node` behavior, unrelated to compaction.
- Latency is effectively unchanged, consistent with the fact that compaction reduces token
  count rather than call count.

## 4. Test 2 — Exercise Recommendation
 
**User message:** `Give me a chest exercise`
 
| Metric | Before (V6) | After (V6.1) | Change |
|---|---:|---:|---:|
| Graph Nodes | history_check_node → coach_node → tools → coach_node | same | — |
| LLM Calls | 2 | 2 | — |
| Tool Calls | 1 (`search_workout_library`) | 1 (`search_workout_library`) | — |
| First LLM Input Tokens | 5,116 | 2,905 | -2,211 |
| First LLM Output Tokens | 219 | 178 | -41 |
| Second LLM Input Tokens | 5,674 | 3,272 | -2,402 |
| Second LLM Output Tokens | 342 | 84 | -258 |
| Total Input Tokens | 10,790 | 6,177 | -4,613 |
| Total Output Tokens | 561 | 262 | -299 |
| **Total Tokens** | **11,351** | **6,439** | **-4,912 (-43.3%)** |
| Latency | 20.18s | 12.79s | -7.39s |
 
**Findings:**
 
- Both LLM calls show a per-call input reduction of approximately 2,200–2,400 tokens,
  consistent with the static baseline savings applying independently to each call in a
  multi-call turn. This confirms that savings compound with call count within a single turn.
- The second call's input remains higher than the first (2,905 → 3,272), reflecting the real
  cost of the `search_workout_library` tool output being appended to context. This is expected
  content cost and was correctly left untouched by compaction.
- The output token reduction on both calls (219→178, 342→84) is most likely attributable to
  normal generation variance (the model is run at temperature 0.4) rather than to compaction,
  since compaction affects only the system prompt and tool schemas, not generation behavior.
- Latency decreased by approximately 37%, a secondary effect of smaller input size.

## 5. Test 3 — Injury + Plan Update
 
**User message:** `My knee hurts when I squat.`
 
| Metric | Before (V6) | After (V6.1) | Change |
|---|---:|---:|---:|
| Graph Nodes | history_check_node → coach_node → tools → coach_node → tools → coach_node → tools → coach_node | history_check_node → coach_node → tools → coach_node | — |
| LLM Calls | 5 | 2 | -3 |
| Tool Calls | 7 (`record_checkin`, `search_fitness_knowledge_base`, `get_current_plan`, `get_past_plans`, `search_workout_library` x2, `update_three_day_plan`) | 2 (`record_checkin`, `get_current_plan`) | -5 |
| First LLM Input Tokens | 5,118 | 2,907 | -2,211 |
| First LLM Output Tokens | 536 | 192 | -344 |
| Second LLM Input Tokens | 6,793 | 3,119 | -3,674 |
| Second LLM Output Tokens | 633 | 105 | -528 |
| **Total Tokens** | **43,662** | **6,323** | **-37,339 (-85.5%)** |
| Latency | 1m 24s | 11.96s | -1m 12s |
 
**Findings:**
 
- This trace is **not a like-for-like comparison** with the original Test 3. The baseline trace
  triggered the full V6 tool workflow (5 LLM calls, 7 tool calls), including
  `search_fitness_knowledge_base`, two `search_workout_library` calls, `get_past_plans`, and
  `update_three_day_plan`. The V6.1 trace called only `record_checkin` and `get_current_plan`
  and stopped, because the existing plan for July 19th already contained an entry labeled
  `knee_friendly_posterior_chain` — the model judged no plan change was required.
- This is a state-driven routing difference (an existing plan entry already matched the
  reported injury), not a compaction effect. Compaction changes token count per call, not
  tool-selection logic. The 85.5% total reduction should not be reported as a compaction
  result, since the two traces do not exercise the same tool path.
- On the two calls that share an equivalent position in the flow, the per-call input reduction
  (-2,211 and -3,674 tokens) is consistent with the static baseline savings observed in Tests 1
  and 2.
- **Limitation:** a valid before/after comparison for this scenario would require rerunning the
  test against a plan state with no existing entry matching the reported injury, so that the
  full multi-tool path is exercised in both traces. This was not performed as part of this
  validation pass.

## 6. Test 4 — Stateful Conversation Follow-up
 
**Conversation:** `Hello` → `How are you?` (second turn measured)
 
| Metric | Before (V6) | After (V6.1) | Change |
|---|---:|---:|---:|
| Graph Nodes | history_check_node → coach_node | history_check_node → coach_node | — |
| LLM Calls | 1 | 1 | — |
| Tool Calls | 0 | 0 | — |
| Input Tokens | 5,238 | 2,924 | -2,314 |
| Output Tokens | 95 | 36 | -59 |
| **Total Tokens** | **5,333** | **2,960** | **-2,373 (-44.5%)** |
| Latency | 10.22s | 5.00s | -5.22s |
 
**Findings:**
 
- This trace involves no tool calls and is directly comparable to the original, with no
  routing differences of the kind observed in Test 3.
- The input reduction (-2,314) tracks the static baseline savings closely; the modest excess
  above the ~2,171-token baseline figure is attributable to one prior conversational turn
  carried in `state["messages"]`, present in both the original and rerun traces.
- This is consistent with the original audit's finding that short multi-turn chat adds only a
  small amount of token growth on top of the static baseline.
- The output token reduction (95 → 36) is attributed to generation variance rather than
  compaction, consistent with the pattern observed in Test 2.
- Latency decreased by approximately 51%.

## 7. Test 6 — Stateful Complex Request (not directly comparable)
 
**Before, conversation:** `Hello` → `How are you?` → `increase the intensity` (agent responds
with clarifying questions, no tool calls) → `check my plan... no equipment... upper body`
*(this final turn is the one measured)*
 
**After, conversation:** `Hello` → `How are you?` → `can you add 1 more exercise on my todays
workout list... increase the intensity` *(this single turn is the one measured)*
 
| Metric | Before (V6) | After (V6.1) |
|---|---:|---:|
| Graph Nodes | history_check_node → coach_node → tools → coach_node → tools → coach_node → tools → coach_node | history_check_node → coach_node → tools → coach_node |
| LLM Calls | 4 | 2 |
| Tool Calls | 4 (`get_current_plan`, `get_past_plans`, `search_workout_library` x2) | 2 (`record_checkin`, `search_workout_library`) |
| Total Tokens | 37,308 | 7,247 |
| Latency | 31.87s | 14.70s |
 
**Findings:**
 
- The two traces do not represent the same conversation. In the baseline trace, the user's
  initial request (`increase the intensity`) produced a clarifying-questions response with no
  tool calls; only the subsequent follow-up message, after the user supplied the requested
  detail, was measured (37,308 tokens).
- In the V6.1 trace, a comparable initial request was resolved directly with tool calls on the
  first attempt, without a clarification round-trip. This is a behavioral improvement, but it
  means the measured turn followed a shorter conversation (three user turns rather than four)
  and used a different, smaller tool path — `record_checkin` and `search_workout_library`,
  versus `get_current_plan`, `get_past_plans`, and two `search_workout_library` calls in the
  original. The existing plan was not consulted in this trace.
- Because conversation length and tool path both differ, the 80.6% total-token difference
  cannot be attributed to compaction. It reflects at least three overlapping factors: static
  baseline reduction, one fewer conversational turn in history, and a different, cheaper
  tool-call path selected by the model.
- A separate finding, unrelated to token usage: the V6.1 trace recommended dumbbell-equipment
  exercises (`Dumbbell Bent-Over Row`, `Dumbbell Shoulder Press`) despite no mention of
  equipment availability in this message. In the original conversation, equipment availability
  ("no equipment") had been stated explicitly. This is a correctness question for the tool
  selection logic, out of scope for the V6.1 token optimization pass, and is noted here for
  future follow-up.
- **Limitation:** a valid before/after comparison for this scenario would require replaying the
  identical conversation script — including the clarification round-trip — against the
  compacted prompt. This was not performed as part of this validation pass.

## 8. Summary
 
| Test | Comparable? | Before Total Tokens | After Total Tokens | Reduction |
|---|---|---:|---:|---:|
| 1 — Small talk | Yes | 5,097 | 2,918 | -42.7% |
| 2 — Exercise recommendation | Yes | 11,351 | 6,439 | -43.3% |
| 3 — Injury + plan update | No (routing differs) | 43,662 | 6,323 | not attributable |
| 4 — Stateful follow-up | Yes | 5,333 | 2,960 | -44.5% |
| 6 — Stateful complex request | No (conversation and routing differ) | 37,308 | 7,247 | not attributable |
 
## 9. Conclusion
 
Across the three directly comparable scenarios (Tests 1, 2, and 4), the static prompt and tool
schema compaction produced a consistent 42.7%–44.5% reduction in total tokens, with per-call
input savings closely matching the predicted ~2,171-token baseline reduction. Savings compound
with the number of LLM calls in a turn, confirming the mechanism identified in the original
audit.
 
Tests 3 and 6 surfaced state- and routing-dependent behavior that changed the tool-call path
between the original and rerun traces, making their totals unsuitable as compaction evidence.
These are documented as known limitations of this validation pass rather than compaction
results. No further reruns are planned for this version; any correction of Tests 3 and 6 is
deferred to a future validation pass if needed.
 
**Status:** V6.1 prompt and tool schema compaction is validated on available data and is ready
to be tagged. Recommended next step: V6.2, addressing conversational context accumulation
(persisted `state["messages"]` growth across turns, identified during review of `agent/graph.py`
and `agent/state.py`).
