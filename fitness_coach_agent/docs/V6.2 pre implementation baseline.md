# V6.2 Pre-Implementation Baseline — Conversational Context Accumulation
 
## 1. Purpose
 
Before implementing view-time message trimming (V6.2 design), this document records baseline
measurements for four purpose-built test scripts. Each isolates a specific accumulation
pattern — workout library results, plan-check results, or knowledge base results — sitting
unused in `state["messages"]` and being resent on a later, unrelated turn.
 
These baselines will be compared against identical reruns after V6.2 is implemented.
 
## 2. Method
 
Each test is a fixed, scripted multi-turn conversation, run once in a fresh session. The
turn of interest (the one measured for comparison) is always the final turn, and is chosen to
be unrelated to the tool calls in the turns before it — isolating whether prior tool output is
being carried forward unnecessarily.

## 3. D3 — Control (accumulation floor)
 
**Script:** fresh session, single turn: `Give me a chest exercise`
 
| Metric | Value |
|---|---:|
| Graph Nodes | history_check_node → coach_node → tools → coach_node |
| LLM Calls | 2 |
| Tool Calls | 1 (`search_workout_library`) |
| First LLM Input Tokens | 2,905 |
| First LLM Output Tokens | 141 |
| Second LLM Input Tokens | 3,272 |
| Second LLM Output Tokens | 54 |
| **Total Tokens** | **6,372** |
| Latency | 17.96s |
 
**Note:** the test design specified "Give me a back exercise" for consistency with D1/D4's
final turn; this run used "chest exercise" instead. The tool-call shape (one
`search_workout_library` call, one `target_area` filter, one `avoid_body_parts` filter) is
equivalent, so this is treated as a valid floor, with a small (~50-token) margin of
imprecision noted rather than corrected.
 
This is the reference floor: the expected first-call input cost for a `search_workout_library`
request with no prior tool history in context.

## 4. D1 — Workout Library Accumulation
 
**Script:**
1. `Give me a chest exercise` → `search_workout_library` called
2. `My knee hurts when I squat.` → `record_checkin` called
3. `Give me a back exercise` *(measured turn)*
| Metric | Value |
|---|---:|
| Graph Nodes | history_check_node → coach_node → tools → coach_node |
| LLM Calls | 2 |
| Tool Calls | 1 (`search_workout_library`) |
| First LLM Input Tokens | 3,485 |
| First LLM Output Tokens | 122 |
| Second LLM Input Tokens | 3,854 |
| Second LLM Output Tokens | 60 |
| **Total Tokens** | **7,521** |
| Latency | 13.95s |
 
**Overhead vs. floor (D3):** first-call input +580 tokens (2,905 → 3,485), attributable to
Turn 1's `search_workout_library` result (5 chest exercises) and Turn 2's `record_checkin`
confirmation still present in `state["messages"]`.

## 5. D2 — Plan Check Accumulation
 
**Script:**
1. `What's on my plan for tomorrow?` → `get_current_plan` called
2. `What about the day after?` → answered directly from Turn 1's result, no new tool call
3. `Did I drink enough water yesterday?` *(measured turn)* → `get_recent_checkins` called
| Metric | Value |
|---|---:|
| Graph Nodes | history_check_node → coach_node → tools → coach_node |
| LLM Calls | 2 |
| Tool Calls | 1 (`get_recent_checkins`) |
| First LLM Input Tokens | 3,137 |
| First LLM Output Tokens | 124 |
| Second LLM Input Tokens | 3,201 |
| Second LLM Output Tokens | 39 |
| **Total Tokens** | **6,501** |
| Latency | 13.46s |
 
**Overhead vs. floor (D3):** first-call input +232 tokens (2,905 → 3,137), smaller than D1,
consistent with `get_current_plan`'s single 3-day-window output being lighter than
`search_workout_library` + `record_checkin` combined.
 
**Behavioral note:** Turn 2 required no new tool call — the model answered from Turn 1's
already-retrieved plan data, since it covered both dates asked about. This is a case where
retained context was used productively rather than wastefully, and is noted as a constraint on
trim timing: trimming must only apply to turns that have already concluded and whose content is
no longer in active use, not proactively to any tool output regardless of continued relevance.

## 6. D4 — Knowledge Base Accumulation
 
**Script:**
1. `Why is protein important for muscle recovery?` → `search_fitness_knowledge_base` called
   **twice** (different queries), returning 8 chunks of raw source text total
2. `Give me a back exercise` *(measured turn)* → `search_workout_library` called
| Metric | Value |
|---|---:|
| Graph Nodes | history_check_node → coach_node → tools → coach_node |
| LLM Calls | 2 |
| Tool Calls | 1 (`search_workout_library`) |
| First LLM Input Tokens | 6,250 |
| First LLM Output Tokens | 446 |
| Second LLM Input Tokens | 7,037 |
| Second LLM Output Tokens | 695 |
| **Total Tokens** | **14,428** |
| Latency | 14.17s |
 
**Overhead vs. floor (D3):** first-call input +3,345 tokens (2,905 → 6,250) — by far the
largest overhead measured. Confirms `search_fitness_knowledge_base` as the single most
expensive tool output to leave unpruned in context: two calls in Turn 1 left 8 full source
chunks (textbook passages with citations) resident in `state["messages"]`, resent in full for
a Turn 2 request that has no relation to nutrition.
 
---
 
## 7. Summary
 
| Test | Turn measured | 1st call input tokens | Overhead vs. floor |
|---|---|---:|---:|
| D3 (control) | fresh session, workout request | 2,905 | — (floor) |
| D2 | Turn 3, unrelated to plan checks | 3,137 | +232 |
| D1 | Turn 3, unrelated to workout/checkin | 3,485 | +580 |
| D4 | Turn 2, unrelated to knowledge base | 6,250 | +3,345 |
 
Overhead scales with the token weight of the tool output(s) left in context, in the order
predicted during design: plan checks (lightest) < workout library results < knowledge base
results (heaviest). This ordering, and the magnitude of D4's overhead in particular, confirms
`search_fitness_knowledge_base` as the highest-priority target for V6.2's trimming logic,
consistent with the disposition table in the V6.2 design document.
 
## 8. Next Steps
 
These four scripts are to be rerun, unchanged, after V6.2's view-time trimming is implemented.
Expected result: each test's measured-turn first-call input tokens should drop toward its
scenario's equivalent floor (a fresh-session call with the same tool-call shape), rather than
carrying the overhead shown above.
